# Decisões de Projeto — 2026-05-06

## Tema

Isolamento correto de ambiente local Firebase Emulator.

---

# Problema Arquitetural

Mesmo com os emuladores Firebase configurados localmente no `.env`, uploads de imagens estavam sendo persistidos na cloud real do Firebase Storage.

O sistema apresentava comportamento híbrido e não determinístico:

- Firestore parcialmente local
- Auth parcialmente local
- Storage persistindo na cloud

Isso gerava:
- risco de poluição de produção
- inconsistência de testes
- perda de confiabilidade no ambiente local
- dificuldade extrema de debugging

---

# Causa Raiz

A principal causa foi a mistura entre:

- `projectId` real do Firebase (`dermasync-3d14a`)
- emuladores locais
- tokens emitidos pelo Auth Emulator
- buckets configurados para cloud real

Além disso, o SDK Python do Firebase Admin + Google Cloud Storage não redirecionava corretamente o endpoint do Storage Emulator apenas via `FIREBASE_STORAGE_EMULATOR_HOST`.

O client interno permanecia usando:

```text
https://storage.googleapis.com ao invés do endpoint local do emulator.
```

Decisões Arquiteturais
1. Separação explícita entre ambiente local e produção

Foi criado um namespace local isolado:

demo-dermasync

Separando completamente:

Ambiente	Project ID
Local	demo-dermasync
Produção	dermasync-3d14a
2. Bootstrap centralizado do Firebase

A inicialização do Firebase Admin SDK foi centralizada em:

app/infra/firebase_app.py

Objetivos:

eliminar múltiplos initialize_app()
evitar side-effects implícitos
garantir ordem determinística de bootstrap
melhorar observabilidade e testabilidade
3. Adapter de Storage deixou de inicializar Firebase

O firebase_storage_adapter deixou de possuir responsabilidade de bootstrap global.

O adapter agora:

apenas consome o app inicializado
não realiza fallback implícito
não executa initialize_app() automaticamente
4. Forçar endpoint explícito do Storage Emulator

Foi necessário configurar explicitamente o endpoint do client do Google Cloud Storage para o emulator local.

Motivo:

o SDK Python não propagava corretamente o emulator endpoint internamente.
Invariantes definidos

As seguintes regras passam a ser obrigatórias:

Nunca usar storage.bucket("bucket-hardcoded")
Nunca inicializar Firebase fora do bootstrap central
Nunca misturar projeto local com projeto cloud
Nunca confiar apenas em env vars sem validar runtime real
Todo ambiente local deve usar namespace demo-*
Observabilidade adicionada

Durante o debugging, tornou-se obrigatório validar explicitamente:

projectId
bucket utilizado
endpoint real (API_BASE_URL)
hosts de emulator carregados

Exemplo validado:

PROJECT=demo-dermasync
BUCKET=demo-dermasync.appspot.com
STORAGE_EMULATOR=http://127.0.0.1:9199
API_BASE_URL=http://127.0.0.1:9199
Aprendizados Técnicos

Este incidente mostrou que:

Firebase Emulator não é automaticamente determinístico
SDKs Firebase e Google Cloud podem divergir internamente
Bootstrap e configuração de ambiente são parte central da arquitetura
Observabilidade explícita é obrigatória em sistemas distribuídos
Impacto Arquitetural

Após esta correção, o ambiente local passou a possuir:

isolamento real de produção
uploads determinísticos
runtime observável
bootstrap explícito
base confiável para testes, workers e pipelines LLM futuros
